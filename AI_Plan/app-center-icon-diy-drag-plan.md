# FlowBoard 应用中心图标 DIY 与拖动改造方案

## 1. 文档目标

1. 为应用中心新增图标 DIY 能力，支持用户自定义图标表现。
2. 为应用中心新增卡片拖动排序能力，支持用户按使用习惯重排。
3. 在不引入重型依赖的前提下，保持现有架构简洁、可维护、可回滚。
4. 输出可直接进入开发阶段的详细实施清单与验收标准。

## 2. 现状调研结论

基于当前代码（`js/app-center.js`、`css/apps.css`、`index.html`）可确认：

1. 当前应用数据结构为数组，存储于 `localStorage` 的 `flowboard_apps`。
2. 应用图标仅支持预设 Font Awesome class + 背景色。
3. 应用列表按数组顺序渲染，无排序字段、无拖拽交互。
4. 应用卡片双击触发启动，卡片顶部有编辑/删除按钮。
5. Electron 已提供通用文件选择能力 `electronAPI.selectFile`，可复用。

结论：基础能力具备，适合在现有模块内增量改造，不需要引入新页面或新服务。

## 3. 需求拆解

### 3.1 功能需求

1. 图标 DIY
   1. 保留现有预设图标选择。
   2. 新增 Emoji 图标模式。
   3. 新增本地图像上传模式（推荐支持 png、jpg、jpeg、webp）。
2. 图标可拖动
   1. 应用卡片可通过拖拽调整顺序。
   2. 拖拽结果持久化，下次打开保持顺序。

### 3.2 约束需求

1. 不引入无必要日志，保持控制台输出简洁。
2. 兼容历史数据，升级后旧用户数据不丢失。
3. 不引入额外后端依赖，保持纯前端 + Electron IPC 结构。

### 3.3 非目标

1. 不做云端同步。
2. 不做图像裁剪编辑器（如裁剪框、旋转、滤镜）。
3. 不做跨页面拖拽（仅应用中心内部排序）。

## 4. 方案选型与决策

### 4.1 图标 DIY 方案对比

| 方案 | 描述 | 优点 | 风险 | 结论 |
|---|---|---|---|---|
| A | 仅支持预设图标扩充 | 实现快，风险低 | 不满足 DIY | 不选 |
| B | 预设 + Emoji | 成本适中，体验提升明显 | 仍缺少图片图标 | 不选 |
| C | 预设 + Emoji + 本地图片上传 | 覆盖完整 DIY 场景 | 需控制存储体积 | 选用 |

选型结论：采用方案 C。

### 4.2 拖动排序方案对比

| 方案 | 描述 | 优点 | 风险 | 结论 |
|---|---|---|---|---|
| A | 引入第三方排序库 | 开发快，交互成熟 | 新依赖、长期维护成本 | 不优先 |
| B | 原生 HTML5 Drag and Drop | 无新增依赖，结构简洁 | 需处理网格排序细节 | 选用 |
| C | Pointer 事件完全自研 | 可统一桌面和触屏 | 复杂度高，测试成本大 | 暂不选 |

选型结论：采用方案 B，移动端先保证降级可用，触屏增强放入后续迭代。

## 5. 推荐实现总览

1. 数据层升级为版本化结构，新增 `order` 与 `iconMeta`。
2. UI 增加图标模式切换区，支持预设、Emoji、上传图片三类。
3. 列表支持拖拽排序，排序结果持久化写回 `localStorage`。
4. 保持旧字段读取兼容，首次加载时自动迁移到新结构。

## 6. 数据模型设计

### 6.1 存储结构

`flowboard_apps` 从旧数组升级为对象：

```json
{
  "version": 2,
  "apps": [
    {
      "id": "app_1740000000000",
      "name": "VS Code",
      "path": "C:/Program Files/Microsoft VS Code/Code.exe",
      "order": 10,
      "iconMeta": {
        "mode": "preset",
        "presetClass": "fas fa-code",
        "emoji": "",
        "imageDataUrl": "",
        "bgColor": "#22c55e",
        "fgColor": "#ffffff"
      },
      "createdAt": "2026-02-25T10:00:00.000Z",
      "updatedAt": "2026-02-25T10:00:00.000Z"
    }
  ]
}
```

### 6.2 字段说明

1. `version`: 存储版本号，当前定义为 2。
2. `order`: 排序权重，数值越小越靠前。
3. `iconMeta.mode`: `preset` | `emoji` | `image`。
4. `iconMeta.imageDataUrl`: 压缩后的 DataURL，仅 `image` 模式使用。
5. `updatedAt`: 编辑、重排后更新时间。

### 6.3 兼容与迁移

兼容旧结构（数组）的迁移规则：

1. 若解析后是数组，按数组索引生成 `order = (index + 1) * 10`。
2. 旧字段 `icon` 映射到 `iconMeta.presetClass`。
3. 旧字段 `color` 映射到 `iconMeta.bgColor`。
4. `iconMeta.mode` 默认设为 `preset`。
5. 迁移成功后立即以 version 2 结构回写存储。

异常处理：

1. JSON 解析失败时返回空结构，不阻断页面。
2. 单个应用记录非法时过滤该条并继续渲染其他项。

## 7. 交互与界面设计

### 7.1 新增交互入口

1. 在应用中心页头新增 `排序模式` 开关按钮。
2. 在添加/编辑应用弹窗中，图标区域改为三模式切换：
   1. 预设图标
   2. Emoji
   3. 上传图片

### 7.2 排序模式交互规则

1. 默认关闭排序模式。
2. 排序模式开启后：
   1. 卡片显示拖拽手柄。
   2. 双击启动临时禁用，防止误触。
   3. 显示轻提示：拖动卡片可调整顺序。
3. 排序模式关闭后恢复双击启动。

### 7.3 图标模式交互规则

1. 预设模式
   1. 复用当前图标库。
   2. 支持背景色选择（保留现有配色逻辑）。
2. Emoji 模式
   1. 输入框限制单个 emoji（按字符簇长度校验）。
   2. 支持背景色与字体色。
3. 上传图片模式
   1. 仅允许 png/jpg/jpeg/webp。
   2. 文件大小限制建议 300KB（原图）；入库前二次压缩。
   3. 支持清空并回退到预设模式。

### 7.4 视觉反馈

1. 拖拽中卡片添加 `is-dragging` 样式（透明度、缩放、阴影）。
2. 目标落点卡片添加 `drop-before` 或 `drop-after` 标识。
3. 上传图标预览统一显示为圆角方形，保持与现有图标容器一致。

## 8. 拖拽排序算法设计

### 8.1 事件模型

1. `dragstart`
   1. 记录 `draggingAppId`。
   2. 设置拖拽数据 `text/plain = appId`。
2. `dragover`（容器级）
   1. `preventDefault()` 允许放置。
   2. 计算当前鼠标最接近卡片及插入方向。
3. `drop`
   1. 计算新顺序。
   2. 重排数组并重算 `order`。
   3. 持久化并重渲染。
4. `dragend`
   1. 清理拖拽状态样式与临时变量。

### 8.2 顺序更新规则

1. 采用全量重排写回，`order` 统一重置为 `10, 20, 30...`。
2. 此策略简单稳定，避免长期多次拖拽造成浮点误差或稀疏值膨胀。

### 8.3 冲突与边界

1. 拖拽到自身位置时不写回。
2. 拖拽目标不存在时不写回。
3. 拖拽中若数据被删除，执行保护退出并重新渲染。

## 9. 图标 DIY 技术方案

### 9.1 图标渲染策略

统一由 `renderAppIcon(app)` 输出 HTML：

1. `preset` -> `<i class="..."></i>`
2. `emoji` -> `<span class="app-icon-emoji">😀</span>`
3. `image` -> `<img src="data:image/...">`

渲染规则：

1. `image` 模式优先显示图片，不再叠加 icon font。
2. `preset/emoji` 仍使用背景色容器。
3. 图片模式容器保留统一圆角与阴影，视觉一致。

### 9.2 上传图片处理流程

1. 用户选择图片文件。
2. 前端校验类型与体积。
3. 使用 `FileReader` 读取原图。
4. 使用 `canvas` 缩放至建议边长 96 或 128。
5. 导出为 webp（quality 0.85）并写入 `imageDataUrl`。
6. 更新预览与缓存草稿状态。

### 9.3 数据体积控制

1. 单图标压缩后目标 <= 50KB。
2. 超过阈值提示用户更换图片。
3. 估算上限：50 个应用约 2.5MB，仍在常见 localStorage 限额内。

## 10. 安全与健壮性设计

1. 图标 class 白名单正则校验：仅允许字母、数字、空格、短横线。
2. 不支持 svg，避免潜在脚本注入与解析差异。
3. 所有用户文本继续走 `escapeHtml`。
4. `img` 仅接受 `data:image/(png|jpeg|webp)` 前缀。
5. 存储写入失败时给出轻提示，不阻断已在内存中的当前编辑。

## 14. 验收标准

1. 应用中心支持三种图标模式：预设、Emoji、图片上传。
2. 应用卡片支持拖动排序，结果可持久化。
3. 旧用户数据不丢失，可自动迁移。
4. 不引入额外第三方拖拽依赖。
5. 控制台不新增无意义日志。

## 15. 风险与回滚

### 15.1 主要风险

1. 图片 DataURL 导致 localStorage 容量接近上限。
2. 拖拽与双击事件冲突导致误启动。
3. 旧数据迁移遗漏边界字段。

### 15.2 缓解措施

1. 上传前后双重体积限制。
2. 排序模式与普通模式隔离，避免并发交互。
3. 迁移函数中保留字段兜底与异常容错。

### 15.3 回滚策略

1. 保留旧字段兼容解析路径。
2. 如需快速回滚，可保留 `version:2` 读取能力但停用新 UI 入口。

## 16. 待你确认的问题

以下问题将影响最终编码实现，建议先确认：

1. 图标 DIY 是否确认包含图片上传能力，还是只做预设+Emoji。
2. 拖动能力是否只要求桌面端，还是要覆盖移动端触屏拖动。
3. 上传图片格式是否允许 svg（本方案默认不支持）。
4. 是否需要增加“恢复默认顺序”按钮。
5. 排序模式是否希望默认常开，还是像本方案一样默认关闭。

## 17. 建议默认决策（若你未特别指定）

1. 采用预设+Emoji+图片上传的完整 DIY。
2. 先交付桌面端拖拽，移动端后续增强。
3. 图片仅支持 png/jpg/jpeg/webp，不支持 svg。
4. 新增恢复默认顺序按钮，便于快速回退。
5. 排序模式默认关闭，避免误触启动。
